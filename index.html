// goroutine 1 renders http{} ────────────────────────────────
http {
    upstream api-backend {
        least_conn;
        server  10.0.0.1:8080  weight=5  max_fails=3;
        server  10.0.0.2:8080  weight=3;
        keepalive  32;
    }
    server {
        listen       80;
        server_name  api.example.com;
        location /api {
            proxy_pass         http://api-backend;
            proxy_http_version 1.1;
            proxy_read_timeout 60s;
        }
    }
}

// goroutine 2 renders stream{} ──────────────────────────────
stream {
    upstream db-pool {
        hash  $remote_addr  consistent;
        server  10.0.0.1:5432  weight=5  max_fails=3;
        server  10.0.0.2:5432  weight=3;
    }
    server {
        listen                  5432;
        proxy_pass              db-pool;
        proxy_connect_timeout   1s;
        proxy_timeout           10m;
        worker_shutdown_timeout 60s; // gracefulDelay
    }
}

// wg.Wait() — both goroutines done, merge + atomic write ────
var wg sync.WaitGroup
wg.Add(2)
go func() { defer wg.Done(); renderHTTP(&httpBlock, c.HTTP) }()
go func() { defer wg.Done(); renderStream(&streamBlock, c.Stream) }()
wg.Wait()

atomicWrite(nginxConfPath, merged) // rename() — atomic
exec.Command("nginx", "-t").Run()   // validate first
exec.Command("nginx", "-s", "reload").Run()

Metrics Pipeline

~1B Events/Day — Zero Regression

Adding L4 introduced three concrete risks to an existing pipeline processing a billion metric points daily. Each was identified early and designed around before any implementation started.

Risk 01
Cardinality Explosion
Operator-named stream server zones → unbounded Clickhouse dimensions → compression destroyed → GROUP BY queries exponentially slower.
Risk 02
Clickhouse Perf Degradation
Adding wide columns to the hot metric table → HTTP dashboard query SLA broken → existing customers impacted before TCP/UDP customers gained anything.
Risk 03
Query Latency Regression
JOIN-based schema approach → existing HTTP dashboards slow under TCP/UDP write load → latency regression without any new feature benefit.
Solution — Schema
Nullable sparse columns
Benchmarked 3 options. Nullable cols on existing table — only populated for stream events. Clickhouse columnar storage makes sparse nulls free. HTTP hot path unchanged.
Solution — Cardinality
5-layer guardrail system
API cap → Receiver LRU guard → __overflow__ dim instead of drop → alert at 80% cap → 7-day rollup MaterializedView (~60x storage reduction).
Solution — Receiver
Dual-path metric model
Nullable protocol and serverZone fields added to base metric event. HTTP events never touch these fields. Stream events populate them. Zero breaking change.
